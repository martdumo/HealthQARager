##  Visión del Proyecto & Caso de Uso

**NEXUS OS** es un **Motor de Razonamiento Reflexivo** diseñado para explorar la estructuración de datos bibliográficos en entornos de investigación. Resuelve un problema crítico en la IA generativa local: *¿Cómo extraer datos técnicos complejos, cruzarlos con variables preexistentes (EHR) y auditar las alucinaciones del modelo usando hardware de consumo?*

La respuesta: **Atomización de tareas, Arquitectura Map-Reduce y Red Teaming (Agentes de Auto-Corrección).**

---

### Por qué el Dominio Médico: Stress-Test de Arquitectura
La elección de la medicina clínica como caso de uso no es fortuita; representa el entorno de mayor complejidad semántica y menor tolerancia a la alucinación, ideal para llevar al límite las capacidades de un sistema RAG local. Durante las pruebas de concepto iniciales, el modelo base exhibió fallas deductivas críticas (ej. sugerir estudios con contraste contraindicados para patologías específicas). 

Para neutralizar esto sin recurrir a reglas *hardcodeadas*, se reestructuró el pipeline implementando un sistema Reflexivo Multi-Agente (Patrón Actor-Critic). Se aislaron dos vectores de escrutinio autónomo:
1. **Revisión de Causas (Auditor Diagnóstico):** Cuestionamiento de la lógica deductiva, búsqueda de sesgos de anclaje y omisión de diagnósticos diferenciales graves.
2. **Revisión de Consecuencias (Auditor de Seguridad):** Validación estricta de iatrogenia, seguridad de estudios propuestos e interacciones farmacológicas cruzadas con la historia clínica (EHR).

Esta arquitectura obliga al sistema a refutar su propio borrador antes de emitir una respuesta, logrando una síntesis final depurada y lógicamente blindada.

##  Arquitectura y Flujo de Información (Actor-Critic RAG)

El sistema está dividido en dos grandes *pipelines*: uno de **Ingesta Offline** (Acelerado por GPU Batching) y otro de **Inferencia Online Reflexiva**.

### 1. Ingesta y Batch Vectorization (ETL Pipeline)

2. Flujo de Razonamiento Multi-Agente (Inference Pipeline)
code Text

[ PACIENTE ] ➜ (Consulta) + (Historia Clínica / EHR)
      │
      ▼
[ AGENTE TRADUCTOR ] ➜ Transforma a Inglés para máxima precisión en el espacio latente.
      │
      ▼
[ MOTOR FAISS ] ➜ Búsqueda Vectorial Paralela (Top 20 por base de datos).
      │
      ▼
[ AGENTES MAP: CAVEMAN ] ➜ Extraen datos telegráficos para no saturar el contexto.
      │
      ▼
[ ORQUESTADOR 1.0 ] ➜ Genera el BORRADOR INICIAL cruzando EHR y Bibliografía.
      │
      ├──▶ [ AGENTE AUDITOR 1: DIAGNÓSTICO ] ➜ Aplica lógica de diagnóstico diferencial. Busca sesgos de anclaje, omisiones de "Red Flags" y evalúa si los síntomas justifican otras patologías más severas no contempladas.
      │
      ├──▶ [ AGENTE AUDITOR 2: SEGURIDAD (FARMACO/IATROGENIA) ] ➜ Aplica validación lógica estricta: cruza el Borrador con el EHR buscando interacciones medicamentosas, alergias, y evalúa la idoneidad de los estudios sugeridos (riesgos de toxicidad, invasividad, o contraindicaciones según patologías crónicas previas).
      │
      ▼
[ META-ORQUESTADOR FINAL ] ➜ Recibe el Borrador + Auditoría 1 + Auditoría 2. Corrige alucinaciones, blinda el texto y emite la SÍNTESIS FINAL NO VINCULANTE.

🔬 Ingeniería de Prompts: Maximizando LLMs Locales

El mayor desafío de correr LLMs en local es la degradación del contexto. NEXUS OS soluciona esto y mitiga el riesgo de responsabilidad civil mediante:
Característica	Naive RAG (Enfoque Tradicional)	NEXUS OS (Multi-Agent Reflexive RAG)
Manejo de Contexto	Inyecta todo el texto crudo en un solo prompt.	Separa en Buckets, el LLM procesa fragmentos cortos con "Caveman Mode".
Mitigación de Riesgos	Asume la salida del LLM como verdad absoluta.	Aplica patrón Actor-Critic. Agentes independientes intentan "romper" y corregir el borrador inicial.
Personalización	Responde genéricamente.	EHR-Aware: Inyecta las variables del paciente (edad, alergias) para el control de seguridad del Auditor 2.
